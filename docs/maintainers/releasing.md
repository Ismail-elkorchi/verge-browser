# Releasing Reference

## One true publish path

Publishing is done through GitHub Actions OIDC workflows:
- `.github/workflows/publish.yml` for release-triggered publish
- `.github/workflows/publish-manual.yml` for manual verification and controlled publish

Token-based registry publishing is intentionally not used in release automation.

## Manual publish workflow

Run `publish-manual.yml` with:
- `ref`: tag, branch, or commit SHA to publish from
- `dry_run`: `true` for verification mode, `false` for real publish

The workflow performs idempotent guards and aborts if the target version already exists on npm or JSR.

## Local verification commands

```bash
npm ci
npm run check:fast
npm run docs:html:jsr
npm pack --dry-run
node scripts/quality/doc-required.mjs
node scripts/release/render-notes.mjs --dry-run
node scripts/release/update-changelog.mjs --dry-run
```

Both publish workflows also upload the generated JSR HTML docs as workflow artifacts so maintainers can inspect the rendered API surface for the exact release candidate that passed the publish gates.

## Release notes format

Release note bullets are generated by `scripts/release/render-notes.mjs` and follow:
- `<PR title> (#123)` where `#123` is a link

Author suffixes are intentionally excluded from generated bullets.

## Changelog updates

Use `scripts/release/update-changelog.mjs` to refresh the `## Unreleased` release-notes block with the same PR bullet format used for release notes.
